# Ch01. 자바스크립트 시작 — 변수 선언과 스코프

## 학습 목표

- 자바스크립트가 어떤 언어인지 큰 그림을 이해한다
- `var` / `let` / `const` 의 차이를 설명할 수 있다
- 스코프(Scope)와 호이스팅(Hoisting)이 무엇인지 이해한다

---

## 1. 자바스크립트란?

웹 페이지를 **살아 움직이게** 만드는 프로그래밍 언어입니다.

```
HTML  →  웹 페이지의 뼈대 (구조)
CSS   →  웹 페이지의 외형 (디자인)
JS    →  웹 페이지의 동작 (버튼 클릭, 데이터 처리 등)
```

> **비유:** HTML이 집의 벽·바닥·지붕이라면, CSS는 벽지·가구 배치 등 인테리어이고,  
> JavaScript는 현관 비밀번호 장치, 자동 조명, 냉난방 제어처럼 **집을 움직이게 하는 것**입니다.

우리가 매일 경험하는 것들이 모두 JavaScript로 만들어집니다:
- 카카오톡 웹: 메시지를 보내면 말풍선이 실시간으로 추가된다
- 쿠팡: 상품을 장바구니에 담으면 숫자가 즉시 올라간다
- 구글 지도: 마우스로 드래그하면 지도가 따라 이동한다
- 유튜브: 동영상을 스크롤하다 보면 관련 영상이 계속 로드된다

### 주요 특징

| 특징 | 설명 |
|------|------|
| **동적 타입** | 변수 선언 시 타입을 명시하지 않아도 됩니다 |
| **인터프리터** | 코드를 한 줄씩 읽으며 바로 실행합니다 |
| **다재다능** | 브라우저, 서버(Node.js), 앱(React Native) 모두 가능 |
| **높은 접근성** | 브라우저 개발자 도구만으로 바로 실행 가능 |

> **동적 타입**이란: Java나 C 같은 언어는 변수를 만들 때 "이건 숫자야", "이건 문자야"를 미리 선언해야 합니다.  
> JavaScript는 `let age = 25;` 처럼 그냥 값을 넣으면 타입을 알아서 파악합니다.  
> 편리하지만, 이 때문에 예상치 못한 오류가 생기기도 합니다. (그래서 TypeScript가 나왔습니다.)

### 라이브러리 vs 프레임워크

둘 다 다른 사람이 미리 만들어 놓은 코드를 가져다 쓰는 것입니다.  
처음에는 비슷해 보이지만 **주도권이 누구에게 있는지**가 다릅니다.

```
라이브러리: "내가 필요할 때 꺼내 쓰는 도구"
  → 개발자가 흐름을 직접 제어
  → 예: jQuery, Lodash

프레임워크: "이미 정해진 틀 안에서 내 코드를 끼워 넣는 것"
  → 프레임워크가 흐름을 제어하고, 개발자는 정해진 방식대로 코딩
  → 예: React, Vue, Angular
```

> **비유:** 라이브러리는 필요할 때 꺼내 쓰는 **도구 상자**(망치, 드라이버 등)입니다.  
> 프레임워크는 **레고 세트**처럼 정해진 조립 방식이 있고, 그 방식대로 조립해야 합니다.

## 2. 변수 선언 — `var` / `let` / `const`

변수는 **데이터를 담아두는 이름표 붙은 상자**입니다.  
JavaScript에는 이 상자를 만드는 방법이 3가지가 있습니다.

### 비교표

| 구분 | `var` | `let` | `const` |
|------|-------|-------|---------|
| 등장 시기 | ES1 (초창기) | ES6 (2015) | ES6 (2015) |
| 스코프 | 함수 스코프 | 블록 스코프 | 블록 스코프 |
| 중복 선언 | ✅ 가능 (위험) | ❌ 불가 | ❌ 불가 |
| 재할당 | ✅ 가능 | ✅ 가능 | ❌ 불가 |
| 호이스팅 | `undefined` 반환 | TDZ → 에러 | TDZ → 에러 |
| 사용 권장 | ❌ 사용 금지 | ✅ | ✅ |

### var의 문제점 (`var.js`)

`var`는 JavaScript가 처음 만들어질 때부터 있던 오래된 방식입니다.  
편의를 위해 만들어졌지만, 오히려 버그를 쉽게 만드는 특성들이 있어  
현재는 **사용하지 않는 것이 원칙**입니다.

```js
// ❌ 문제 1: 중복 선언 허용 → 실수를 잡아주지 못함
var userName = 'John';
var userName = 'Doe'; // 에러 없이 덮어씀
// 코드가 수천 줄이라면, 이미 선언된 변수인지 모르고 실수로 덮어쓸 수 있습니다

// ❌ 문제 2: 블록 스코프 없음 → if/for 블록 탈출
if (true) {
  var age = 20;
}
console.log(age); // 20 — 블록 밖에서도 접근됨!
// 의도치 않게 전 세계 어디서든 age에 접근 가능

// ❌ 문제 3: 호이스팅으로 undefined 반환
console.log(score); // undefined (에러 아님 — 이게 더 위험!)
var score = 100;
```

### let과 const 사용법 (`let_const.js`)

```js
// ✅ let: 재할당 가능, 중복 선언 불가
let count = 0;
count = 1; // OK
// let count = 2; // SyntaxError! → 실수를 즉시 알아챌 수 있어 안전

// ✅ const: 재할당 불가 (상수)
const PI = 3.14;
// PI = 4; // TypeError! → 이 값은 절대 바뀌면 안 된다는 의도 표현

// ✅ const + 객체: 내용 변경은 가능
const user = { name: 'Kim' };
user.name = 'Lee'; // OK — 객체 내부 속성 변경은 가능
// user = {};        // TypeError — 변수 자체가 가리키는 객체를 바꾸는 것은 불가
```

> **const와 객체:** const는 변수가 가리키는 **주소**를 바꾸지 못하게 합니다.  
> 주소 안에 있는 내용물을 바꾸는 것은 허용됩니다.  
> 비유: 주소가 적힌 종이(const)가 있을 때, 그 종이에 다른 주소를 쓸 수 없습니다.  
> 하지만 그 주소로 찾아가서 집 안의 가구를 바꾸는 것은 가능합니다.

### 선택 기준

```
1순위: const  →  기본적으로 const 사용 (불변 의도 명시)
2순위: let    →  값이 바뀌어야 할 때만 let 사용
사용 금지: var  →  구형 코드에서만 볼 것, 직접 쓰지 않기
```

> **왜 const를 기본으로 쓸까요?**  
> "이 값은 바뀌지 않는다"는 의도를 코드에 표현할 수 있습니다.  
> 나중에 코드를 읽는 사람(미래의 나 포함)이 "이 변수는 안전하다"고 확신할 수 있습니다.

## 3. 스코프 (Scope)

**스코프(Scope)** 는 변수가 **어디서 접근 가능한지**의 유효 범위입니다.

> **비유:** 회사 건물에 비유할 수 있습니다.  
> - **전역 스코프** = 회사 전체 공용 구역 (로비, 화장실) — 누구나 접근 가능  
> - **함수 스코프** = 각 부서 사무실 — 해당 부서 직원만 접근  
> - **블록 스코프** = 개인 파티션 공간 — 해당 공간의 사람만 접근

```
전역 스코프: 코드 어디서든 접근 가능
│
└── 함수 스코프: 함수 내부에서만 접근 가능 (var)
│       function fn() {
│         var x = 1;  ← 함수 밖에서 접근 불가
│       }
│
└── 블록 스코프: {} 블록 내부에서만 접근 가능 (let, const)
        if (true) {
          let y = 2;  ← 블록 밖에서 접근 불가
          const z = 3;
        }
```

### 스코프 체인

안쪽에서 바깥쪽 변수에는 접근할 수 있습니다. 반대는 불가합니다.

> **비유:** 방 안에서는 방 밖을 볼 수 있지만, 방 밖에서는 방 안을 볼 수 없습니다.  
> (방 문이 닫혀 있다고 상상하세요)

```js
const outer = '바깥';

function fn() {
  const inner = '안쪽';
  console.log(outer); // OK — 바깥 변수 접근 가능 (방 안에서 밖 보기)
}

// console.log(inner); // ReferenceError — 안쪽 변수는 밖에서 접근 불가 (방 밖에서 안 못 보기)
```

## 4. 호이스팅 (Hoisting)

자바스크립트 엔진이 코드를 실행하기 **전에**, 변수·함수 선언을 스코프 맨 위로 **끌어올리는** 동작입니다.

> 실제 코드가 이동하는 게 아니라, 엔진이 그렇게 처리한다는 의미입니다.

> **비유:** 수업 시작 전에 선생님이 교실에 들어와서 출석부를 훑어보는 것과 같습니다.  
> "이 수업에는 철수, 영희, 민준이 있겠구나"를 먼저 파악(호이스팅)하고,  
> 그다음 수업(코드 실행)을 시작합니다.

### 함수 선언문 — 전체가 호이스팅

함수 선언문은 함수 전체가 맨 위로 끌어올려지기 때문에 선언 전에 호출해도 작동합니다.

```js
sayHello(); // ← 선언 전인데 정상 작동!

function sayHello() {
  console.log('Hello!');
}
```

### var — 선언만 호이스팅, 값은 undefined

`var`는 선언 부분만 위로 올라가고, 값 초기화(`= 100`)는 제자리에 남습니다.  
그래서 선언 전에 접근하면 `undefined`가 반환됩니다. (에러가 아닌 것이 더 위험!)

```js
// 실제 동작 순서:
// var score;               ← 선언이 맨 위로 끌어올려짐 (호이스팅)
// console.log(score);      → undefined
// score = 100;             ← 초기화는 제자리

console.log(score); // undefined (에러 아님!)
var score = 100;
console.log(score); // 100
```

### let / const — TDZ (Temporal Dead Zone)

`let`과 `const`도 호이스팅은 되지만, **TDZ(일시적 사각지대)** 구간에서 접근하면 즉시 에러가 발생합니다.  
이것이 오히려 **더 안전**합니다. 선언 전에 접근했다는 실수를 즉시 알 수 있으니까요.

```
코드 시작
    │
    ▼  ← TDZ 시작 (호이스팅은 되었지만 접근 불가)
    │  console.log(city); → ReferenceError!
    │
    ▼  let city = 'Seoul'; ← 선언문 도달 → TDZ 종료
    │
    ▼  console.log(city); → 'Seoul' (정상 접근)
```

### 함수 표현식 — 변수 호이스팅 규칙을 따름

함수를 변수에 담는 방식(함수 표현식)은 **변수의 호이스팅 규칙**을 따릅니다.

```js
// printBye(); // TypeError (var이므로 선언 전엔 undefined, 호출 불가)
var printBye = function () { console.log('Bye!'); };

// greetUser(); // ReferenceError (const이므로 TDZ)
const greetUser = function () { console.log('안녕!'); };
```

### 호이스팅 요약

| 선언 방식 | 호이스팅 | 선언 전 접근 |
|----------|---------|------------|
| `var` 변수 | O | `undefined` |
| `let` 변수 | O (TDZ) | `ReferenceError` |
| `const` 변수 | O (TDZ) | `ReferenceError` |
| 함수 선언문 | O (전체) | 정상 작동 |
| 함수 표현식 (`var`) | O | `TypeError` |
| 함수 표현식 (`const`) | O (TDZ) | `ReferenceError` |
