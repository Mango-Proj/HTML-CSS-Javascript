# ch01 · 브라우저 렌더링 & 이벤트

브라우저가 화면을 그리는 원리와 자바스크립트 이벤트 처리 방식을 학습하는 예제입니다.  
`index.html`을 브라우저에서 열면 모든 개념을 직접 실행하며 확인할 수 있습니다.

파일을 직접 브라우저에서 열면 됩니다. (별도 서버 불필요)

```
ch01_browser/index.html  →  브라우저로 열기
```

## 핵심 개념 요약

### 1. 브라우저 렌더링 프로세스 (6단계)

#### 렌더링이란?

우리가 브라우저 주소창에 URL을 입력하면, 서버에서 HTML·CSS·JS 파일을 받아옵니다.  
브라우저는 이 파일들을 읽어서 우리가 보는 **화면(픽셀)**으로 변환합니다.  
이 과정 전체를 **렌더링(Rendering)** 이라고 합니다.

> **왜 렌더링 원리를 알아야 하나요?**  
> 렌더링 과정을 이해하면 "왜 이 코드가 페이지를 느리게 만드는지",  
> "왜 애니메이션이 뚝뚝 끊기는지"를 이해하고 최적화할 수 있습니다.  
> 같은 결과를 만들더라도 렌더링 비용이 10배 이상 차이나는 방식이 존재합니다.

#### 파싱(Parsing)이란?

브라우저가 HTML 파일을 받으면 코드를 **위에서 아래로 한 줄씩 읽으며 의미를 해석**하는 작업을 합니다.  
이 과정을 **파싱(Parsing)** 이라고 합니다.

> **비유:** 악보(HTML 코드)를 눈으로 읽으며 머릿속에서 음악(웹 페이지 구조)을 구성하는 것과 같습니다.

#### 렌더링 6단계

웹 페이지가 화면에 표시되기까지 브라우저가 수행하는 순서입니다.

| 단계 | 이름 | 쉬운 설명 |
|------|------|-----------|
| 1 | **HTML 파싱 → DOM** | HTML 코드를 읽어 페이지의 **뼈대(구조)** 를 만듭니다 |
| 2 | **CSS 파싱 → CSSOM** | CSS 코드를 읽어 **스타일 규칙 목록** 을 만듭니다 |
| 3 | **렌더 트리 생성** | 뼈대 + 스타일 규칙을 합칩니다. 실제로 화면에 보이는 요소만 포함합니다 (`display:none` 제외) |
| 4 | **Layout (리플로우)** | 각 요소가 **어디에, 얼마나 크게** 배치될지 계산합니다 |
| 5 | **Paint (리페인트)** | 계산된 위치에 **색상·이미지·텍스트 등을 픽셀로 그립니다** |
| 6 | **Composite** | 여러 레이어를 **하나로 합쳐** 최종 화면을 완성합니다 |

> **DOM / CSSOM 이란?**
> - **DOM** (Document Object Model): HTML의 각 요소(`<div>`, `<p>` 등)를 자바스크립트로 조작할 수 있는 트리(나무) 구조로 변환한 것입니다.
> - **CSSOM** (CSS Object Model): CSS 규칙들을 브라우저가 빠르게 처리하기 위해 변환한 구조입니다.

> **비유로 전체 흐름 이해하기:**  
> HTML = 건물 설계도(구조), CSS = 인테리어 디자인 계획서, JS = 동작 명세서  
> 1. 설계도 읽기 → 2. 인테리어 계획 읽기 → 3. 두 가지 합쳐서 실제 건물 그림 그리기  
> → 4. 각 방의 위치·크기 계산 → 5. 벽지·가구 색칠 → 6. 여러 층 합쳐서 완성

### 2. 리플로우 vs 리페인트 vs 컴포지트

렌더링 비용이 어느 단계에서 발생하는지에 따라 성능 차이가 납니다.  
같은 시각적 결과를 만들더라도 **어떤 CSS 속성을 사용하느냐**에 따라  
브라우저가 1단계부터 다시 계산할 수도 있고, 마지막 단계만 실행할 수도 있습니다.

#### 🏗️ 리플로우 (Reflow / Layout)

요소의 **기하학적인 수치(너비, 높이, 위치)**가 변경될 때 브라우저가 레이아웃을 다시 계산하는 과정입니다.

- **특징**: 가장 비용이 많이 드는 작업입니다. 한 요소의 리플로우가 발생하면 자식·주변 요소까지 **연쇄적으로** 리플로우가 일어날 수 있습니다.
- **발생 조건**:
  - DOM 노드의 추가·제거
  - 요소의 크기·위치 변경 (`width`, `height`, `margin`, `padding`, `top`, `left` 등)
  - 윈도우 리사이징
  - 폰트 변경, 이미지 크기 변경

> **비유:** 사무실에서 책상 하나를 옮기면 주변 책상 위치도 전부 다시 재배치해야 하는 상황과 같습니다.  
> 책상 하나를 움직였을 뿐인데 옆 책상, 앞 책상까지 모두 다시 정리해야 합니다.

#### 🎨 리페인트 (Repaint / Paint)

리플로우가 끝난 뒤, 계산된 수치를 바탕으로 **화면에 실제 픽셀을 채워 넣는(그리는)** 과정입니다.

- **특징**: 요소의 크기·위치는 변하지 않고 시각적인 외형만 바뀔 때 발생합니다. 리플로우보다는 가볍지만 CPU/GPU 자원을 소모합니다.
- **발생 조건**: `background-color`, `color`, `visibility`, `outline`, `box-shadow`, `border-radius` 등 시각적 속성 변경
- **주의**: 리플로우가 발생하면 리페인트는 **반드시 따라옵니다** (위치를 계산해야 그릴 수 있으므로).

> **비유:** 책상은 그대로인데 위에 올려둔 물건 색깔만 바꾸는 작업입니다.  
> 재배치는 필요 없지만, 다시 그려야 합니다.

#### 🧩 컴포지트 (Composite)

화면을 여러 개의 **레이어(Layer)**로 나누어 그린 뒤 이를 하나로 합쳐 최종 화면을 만드는 단계입니다.

- **특징**: 리플로우와 리페인트를 **건너뛰고** 컴포지트만 발생시키는 것이 성능 최적화의 핵심입니다. GPU가 담당하므로 CPU 부담이 없고 매우 부드러운 애니메이션을 제공합니다.
- **발생 조건**: `transform`(이동·회전·크기 조절), `opacity`(투명도)
- **이점**: 이미 그려진 레이어를 위치만 옮기거나 겹치기 때문에 처리 속도가 압도적으로 빠릅니다.

> **비유:** 투명 필름 여러 장에 각각 그림을 그려 두고, 필름 자체를 이동시키는 것과 같습니다.  
> 그림을 다시 그릴 필요가 없습니다.  
> 유튜브에서 영상이 부드럽게 재생되는 것, 앱의 애니메이션이 60fps로 동작하는 것이 이 원리 덕분입니다.

#### 비용 비교

| 종류 | 유발 속성 예시 | 실행 단계 | 비용 |
|------|---------------|-----------|------|
| **리플로우** | `width`, `height`, `margin`, `top` | Layout → Paint → Composite | 🔴 높음 |
| **리페인트** | `color`, `background`, `border-color` | Paint → Composite | 🟡 중간 |
| **컴포지트 전용** | `transform`, `opacity` | Composite 만 | 🟢 낮음 |

**최적화 전략:**

```js
// ❌ 리플로우 2번 유발 (스타일 속성을 각각 변경)
el.style.width  = '200px';
el.style.height = '100px';

// ✅ 클래스 하나만 변경 → 렌더링 연산 최소화
el.classList.toggle('active');

// ✅ transform 사용 → Layout·Paint 생략, Composite만 실행
// 요소를 이동시킬 때 top/left 대신 transform을 사용하세요
el.style.transform = 'translateX(200px)';
```

> **실무 팁:** 버튼 클릭 애니메이션, 슬라이드 전환, 드래그 효과 등을 만들 때  
> `top`, `left` 대신 `transform: translate()`를 사용하면 훨씬 부드러운 결과를 얻을 수 있습니다.

### 3. DocumentFragment — 리플로우 최소화

#### 문제: DOM에 직접 추가하면 왜 느릴까?

브라우저는 자바스크립트로 DOM에 요소를 추가할 때마다 **"화면을 다시 그려야 하나?"** 를 판단합니다.  
요소를 하나 추가할 때마다 레이아웃 계산(리플로우)이 발생할 수 있기 때문에,  
100개 요소를 하나씩 추가하면 리플로우가 최대 **100번** 일어날 수 있습니다.

```js
// ❌ 문제: 루프마다 DOM에 직접 삽입 → 리플로우가 반복 발생
for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `항목 ${i}`;
  list.appendChild(li);  // 삽입할 때마다 브라우저가 레이아웃을 재계산할 수 있음
}
```

#### 해결책: DocumentFragment

`DocumentFragment`는 **메모리 안에만 존재하는 임시 컨테이너**입니다.  
실제 DOM(화면)과 연결되어 있지 않기 때문에, 아무리 많은 요소를 추가해도 리플로우가 발생하지 않습니다.  
조립이 끝난 뒤 Fragment를 실제 DOM에 **단 한 번** 삽입하면 리플로우도 딱 한 번만 일어납니다.

> **비유:** 식당에서 웨이터가 요리를 하나씩 나를 때마다 주방에서 홀까지 왔다 갔다 하는 것보다,  
> **쟁반(Fragment)에 요리를 모두 올려두고 한 번에 나르는 것**이 훨씬 효율적입니다.

```js
// ✅ 해결: Fragment에 모두 조립 후, 한 번에 DOM에 삽입

// 1. 쟁반(Fragment)을 준비합니다 — 메모리에만 존재, 화면과 무관
const fragment = document.createDocumentFragment();

for (let i = 0; i < 100; i++) {
  const li = document.createElement('li');
  li.textContent = `항목 ${i}`;
  fragment.appendChild(li);  // 쟁반에 올리기 (리플로우 없음)
}

// 2. 쟁반째로 DOM에 한 번에 올립니다 → 리플로우 단 1회
list.appendChild(fragment);
```

#### 정리

| 방식 | 동작 | 리플로우 횟수 |
|------|------|:---:|
| 직접 추가 (`appendChild` 반복) | 요소마다 DOM 업데이트 | 최대 N번 |
| **Fragment 사용** | 메모리에서 조립 → 한 번에 삽입 | **1번** |

### 4. 스크립트 로딩 — async vs defer

#### 문제: 스크립트가 페이지 로딩을 막는다

브라우저는 HTML을 위에서 아래로 읽어 내려가다가 `<script>` 태그를 만나면 **HTML 파싱을 즉시 중단**합니다.  
스크립트 파일을 다운로드하고 실행이 끝날 때까지 나머지 HTML을 읽지 않기 때문에,  
네트워크가 느리거나 스크립트가 크면 **사용자에게 아무것도 표시되지 않는 빈 화면**이 이어질 수 있습니다.

> **비유:** 책을 읽다가 모르는 단어를 발견했을 때, 사전을 찾아보는 동안 책 읽기를 완전히 멈추는 것과 같습니다.  
> 사전이 두껍고 단어를 찾는 데 오래 걸릴수록 독서가 더 오래 멈춥니다.

```html
<!-- ❌ 이 위치에 있으면 HTML 파싱이 중단됩니다 -->
<head>
  <script src="app.js"></script>  <!-- 다운로드·실행 동안 아무것도 표시 안 됨 -->
</head>
```

#### 3가지 로딩 방식 비교

**`<script>` (기본 — 비권장)**

```
HTML 파싱 ──────► ⛔ 중단
                   └─► JS 다운로드 ─► JS 실행 ─► HTML 파싱 재개 ──► 완료
```

- HTML 파싱 중 스크립트를 만나면 **즉시 멈춥니다.**
- 스크립트 다운로드 + 실행이 끝나야만 파싱을 재개합니다.
- 스크립트가 길거나 느리면 사용자는 **빈 화면을 오래 봐야 합니다.**

---

**`async` (비동기 — 독립 스크립트용)**

```
HTML 파싱 ──────────────────────────────► ⛔ 잠깐 중단 ─► 파싱 재개 ──► 완료
              ↳ JS 다운로드 (동시 진행) ──────────────►
                                          └─► JS 실행
```

- HTML 파싱과 JS 다운로드가 **동시에** 진행됩니다.
- 다운로드가 끝나는 즉시 파싱을 잠깐 멈추고 스크립트를 실행합니다.
- **실행 순서가 보장되지 않습니다.** (먼저 다운로드된 것부터 실행)
- 다른 스크립트에 의존하지 않는 독립적인 기능(광고, 통계 등)에 적합합니다.

---

**`defer` (지연 — 일반 앱 스크립트 권장)**

```
HTML 파싱 ──────────────────────────────────────────────────► 완료
              ↳ JS 다운로드 (동시 진행) ──────────────────►
                                                              └─► JS 실행
```

- HTML 파싱과 JS 다운로드가 **동시에** 진행됩니다.
- 스크립트 실행은 **HTML 파싱이 완전히 끝난 뒤**에 시작됩니다.
- 여러 `defer` 스크립트가 있어도 **HTML에 작성된 순서대로** 실행됩니다.
- DOM 요소가 모두 준비된 상태에서 스크립트가 실행되므로, `document.querySelector()` 등이 안전합니다.

> **비유:**
> - `<script>` : 요리 중에 레시피가 필요하면 **요리를 멈추고** 레시피를 찾아 읽은 뒤 다시 시작합니다.
> - `async` : 요리하면서 **레시피를 동시에 검색**하고, 찾아지는 순간 요리를 잠깐 멈추고 읽습니다.
> - `defer` : 요리하면서 **레시피를 동시에 검색**하되, 요리가 **완성된 후에** 레시피를 확인합니다.

#### 비교 요약

| 구분 | 병렬 다운로드 | 실행 시점 | 순서 보장 | 권장 용도 |
|------|:---:|---|:---:|---|
| `<script>` | ❌ | 태그를 만나는 즉시 | ✅ | (비권장) |
| `async` | ✅ | 다운로드 완료 직후 | ❌ | 광고·분석 등 독립 스크립트 |
| `defer` | ✅ | HTML 파싱 완료 후 | ✅ | **일반 앱 스크립트 (권장)** |

#### 권장 사용법

```html
<!-- ✅ defer: 파싱을 막지 않고, DOM이 완성된 후 실행 -->
<head>
  <script defer src="app.js"></script>
</head>

<!-- ✅ body 끝에 배치하는 방법도 유사한 효과 (전통적인 방법) -->
<body>
  ...
  <script src="app.js"></script>  <!-- 대부분의 HTML이 파싱된 후 실행 -->
</body>
```

> **`defer`를 `<head>`에 쓰는 것이 권장되는 이유:**  
> 브라우저가 HTML을 읽자마자 다운로드를 시작하므로 `</body>` 직전에 쓰는 것보다 **더 일찍 다운로드가 끝납니다.**  
> 실행은 파싱 완료 후이므로 DOM 접근도 안전합니다.

### 5. 이벤트 전파 — 버블링 & 캡처링

#### 이벤트가 "전파"된다는 것은 무슨 의미인가?

웹 페이지의 요소들은 **부모-자식 관계(트리 구조)**로 중첩되어 있습니다.  
버튼이 div 안에 있고, 그 div가 또 다른 div 안에 있는 것처럼요.  
사용자가 어떤 요소를 클릭하면, 그 이벤트는 트리를 따라 이동합니다.  
이것을 **이벤트 전파(Event Propagation)** 라고 합니다.

이벤트는 3단계를 거칩니다.

```
[1단계: 캡처링] document에서 클릭된 요소 방향으로 내려갑니다
document
  └─ body
       └─ outer      ↓ 내려감 (캡처링)
            └─ middle ↓
                 └─ inner ← [2단계: 타깃] 실제로 클릭된 요소

[3단계: 버블링] 클릭된 요소에서 document 방향으로 올라갑니다
document
  └─ body
       └─ outer      ↑ 올라감 (버블링)
            └─ middle ↑
                 └─ inner ← 이벤트 발생
```

> **비유:** 물 속에서 거품(bubble)이 아래에서 위로 올라오는 것처럼, 이벤트도 자식 → 부모 방향으로 올라갑니다.

| 단계 | 방향 | 기본 동작 |
|------|------|-----------|
| **캡처링** | `document` → 클릭된 요소 (내려감) | 기본적으로 리스너가 반응하지 않음 |
| **타깃** | 클릭된 요소 자체 | 이벤트 발생 |
| **버블링** | 클릭된 요소 → `document` (올라감) | 기본값 — 리스너가 반응함 |

```js
// 기본: 버블링 단계에서 실행됩니다
el.addEventListener('click', handler);

// 캡처링 단계에서 실행하려면 세 번째 인자에 { capture: true }를 전달합니다
el.addEventListener('click', handler, { capture: true });

// 버블링 중단: 이 요소 이후로 이벤트가 더 이상 올라가지 않습니다
el.addEventListener('click', (e) => {
  e.stopPropagation();
  // 이 요소에서 클릭 처리 완료, 부모 요소로 전파 차단
});
```

> **버블링이 왜 중요한가?**  
> 버블링 덕분에 **이벤트 위임(Event Delegation)** 이 가능합니다.  
> 자식 요소 100개에 각각 이벤트를 달지 않고, 부모 하나에만 달아서 모두 처리할 수 있습니다.

### 6. 이벤트 위임 (Event Delegation)

#### 문제: 요소마다 리스너를 다는 방식의 한계

버튼이 100개 있다면 각 버튼에 클릭 리스너를 100개 달아야 할까요?  
또한 나중에 버튼이 **동적으로 추가**되면, 기존 리스너가 없어서 클릭해도 아무 반응이 없습니다.

```js
// ❌ 문제점 두 가지:
// 1. 리스너를 요소 개수만큼 등록해야 합니다 (메모리 낭비)
// 2. 나중에 추가된 li는 리스너가 없어서 클릭이 동작하지 않습니다
document.querySelectorAll('li').forEach(li => {
  li.addEventListener('click', handler);
});
```

#### 해결책: 이벤트 위임

**버블링**을 활용합니다. 자식을 클릭했을 때 이벤트가 부모까지 올라온다는 특성을 이용해,  
**부모 요소 하나에만 리스너를 달면** 모든 자식의 이벤트를 한 곳에서 처리할 수 있습니다.

> **비유:** 회사에서 각 직원에게 전화기를 주는 것보다, **안내 데스크 하나**에서 모든 문의를 받아 담당자에게 연결해 주는 것과 같습니다.  
> 새 직원이 입사해도(새 요소가 추가되어도) 안내 데스크는 변함없이 처리합니다.

```js
// ✅ 부모 ul에 리스너 하나만 등록
ul.addEventListener('click', (e) => {
  // e.target: 실제로 클릭된 요소
  // closest(): 자신 또는 가장 가까운 조상 중 조건에 맞는 요소를 찾습니다
  if (e.target.tagName === 'LI') {
    handler(e.target);
  }
});
// → 나중에 li가 동적으로 추가되어도 자동으로 처리됩니다!
```

`data-action` 속성 패턴으로 여러 버튼을 하나의 리스너에서 분기할 수 있습니다.

```js
// 완료 버튼: <button data-action="done">
// 삭제 버튼: <button data-action="delete">
// → 리스너는 하나, 처리는 data-action 값으로 분기

list.addEventListener('click', (e) => {
  // closest('[data-action]'): data-action 속성을 가진 가장 가까운 부모/자신을 찾음
  const btn = e.target.closest('[data-action]');
  if (!btn) return;  // data-action이 없는 곳을 클릭하면 무시

  if (btn.dataset.action === 'done')   { /* 완료 처리 */ }
  if (btn.dataset.action === 'delete') { /* 삭제 처리 */ }
});
```

> **`closest()`란?**  
> 클릭된 요소 자신부터 시작해서 부모 방향으로 올라가며 조건에 맞는 요소를 찾습니다.  
> 버튼 안에 아이콘(`<i>`)이 있을 때, 아이콘을 클릭해도 버튼을 찾을 수 있어서 매우 유용합니다.

#### 이벤트 위임이 유리한 상황

| 상황 | 이유 |
|------|------|
| 동적으로 요소가 추가·삭제되는 목록 | 추가될 때마다 리스너를 새로 달 필요 없음 |
| 같은 종류의 버튼이 수십 개 이상인 경우 | 리스너 수를 1개로 줄여 메모리 절약 |
| 할 일 목록, 댓글, 상품 목록 등 | 데이터 기반으로 동적으로 렌더링하는 UI |
